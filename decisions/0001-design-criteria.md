# ADR-0001: 設計判断基準

- **Date**: 2026-05-23
- **Status**: Accepted
- **Context**: ATF はプロトコル設計プロジェクト。多数の trade-off が発生するため、判断の一貫性を保つために優先順位を固定する。

## 決定

以下の 8 つの基準を **数字が小さいほど優先** として扱う。trade-off で迷ったらこの順序で判定する。

| 優先 | 基準 | 競合時の判定例 |
|---|---|---|
| 1 | **採用障壁の低さ** | DID 必須化 ❌ → URI + JWK URI ✅（広く使われている） |
| 2 | **既存標準の借用** | 独自 manifest schema ❌ → A2A Agent Card ✅ |
| 3 | **Federation の純度** | 中央 registry 前提 ❌ → cross-signed events ✅ |
| 4 | **MVP の最小性** | 全 10 層実装 ❌ → L0+L1min+L4+L6min+L7 ✅ |
| 5 | **Vendor 中立性** | 単一企業 SaaS ❌ → OSS Apache 2.0 ✅ |
| 6 | **セキュリティの省略不能性** | 署名 opt-out 可 ❌ → core 操作で必須 ✅ |
| 7 | **互換性保証** | breaking change ❌ → 2 minor 後方保証 ✅ |
| 8 | **プライバシー既定** | payload を audit に保存 ❌ → hash のみ ✅ |

## 適用ガイド

- 設計判断のたびに「この選択は基準 N に対してどうか」を明示する
- 基準 1 と 6 が矛盾する場合（=採用を取るためにセキュリティを緩める）は **基準 6 優先**
- 基準 4（MVP 最小性）はスコープクリープ防止。「便利だから入れる」は不可
- 新たに別基準を加える必要が出たら、本 ADR を更新するか別 ADR を作って参照

## 反例（ここから外れる場合の例外条件）

- セキュリティ上の脆弱性発見 → 基準 6 を超えて即修正、基準 7（互換性）は犠牲にしてよい
- IETF / W3C 等が新標準を採択 → 基準 2 に従い独自実装より優先採用
- パフォーマンスが「使い物にならない」レベルの問題 → 例外として性能改善のみ実施可
